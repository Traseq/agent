This guide is for developers building AI agents that create trading strategies
in Traseq. You'll learn how to go from generating a strategy payload to
running a backtest — all through the Public Agent API.

## Before you start

Public Agent API calls require a workspace API key, not a wallet private key or
exchange secret. Create one in the Traseq app from **Settings > API Keys**,
copy the plaintext key while it is shown, and store it in your secret manager
or runtime environment as `TRASEQ_API_KEY`.

Every plan can create API keys and run the core agent workflow. Free and Plus
agents are intentionally constrained by lower key quotas, research credits,
strategy limits, saved-result limits, and related workspace capacity.

Do not paste API keys into AI prompts or expose them to the model. Your agent
host should inject the key into the `x-api-key` HTTP header outside the model's
tool input.

For a write-capable strategy agent, create a `member` key with only the scopes
needed for the workflow:

- `workspace_read`
- `strategies_read`
- `strategies_write`
- `backtests_read`
- `backtests_write`

Add `system_strategies_read` only when the agent needs to inspect system
templates.

## Strategy Authoring Contract

The Public Agent API accepts **SignalGraph v2** for strategy authoring. Your
agent should submit `signalGraph` plus `settings`; internal execution formats
are not part of the public write contract.

## Minimum scopes

For an autonomous agent that validates, writes strategies, and runs backtests,
the recommended minimum scopes are:

- `workspace_read`
- `strategies_read`
- `strategies_write`
- `backtests_read`
- `backtests_write`

Recommended add-ons:

- `system_strategies_read`
  Useful if your agent starts from built-in templates.

## Recommended request sequence

1. `GET /public/v1`
   Read the manifest and discover the workflow contract.
2. `GET /public/v1/workspace`
   Inspect workspace identity, API key scopes, and subscription tier.
3. `GET /public/v1/usage`
   Check current budget, usage, and hard limits before write or backtest workflows.
4. `GET /public/v1/capabilities`
   Fetch machine-readable indicators, indicator args, node fields, strategy
   bindings, risk authoring metadata, operators, tier limits, and validation
   issue shape before authoring a payload.
5. `POST /public/v1/strategies/validate`
   Validate the draft payload after local assembly/preflight and repair it until
   `valid === true`.
6. `POST /public/v1/strategies`
   Create a draft strategy.
7. `POST /public/v1/strategies/:id/versions/finalize`
   Promote a version to `ready`.
8. `POST /public/v1/backtests`
   Queue a backtest.
9. `GET /public/v1/backtests/:id`
   Poll until the backtest reaches a terminal state.

## Discovering capabilities

Always call `GET /public/v1/capabilities` before your agent writes a strategy.
This endpoint returns everything your agent needs to know about what's
supported:

- available indicators and their `args`
- supported node kinds, node fields, strategy bindings, and risk metadata for
  SignalGraph
- operators, enums (timeframes, market fields, patterns)
- plan-specific limits
- the validation issue format (`code`, `path`, `field`, `message`, `severity`)

Use this as your source of truth. Do not guess indicator names, `args`, node
fields, bindings, risk paths, or operators from prompts alone.

## Structured validation issues

`POST /public/v1/strategies/validate` returns machine-readable issues. Graph
validation issues and 400 request schema failures use the same flat `issues[]`
shape. Request schema failures also include `errorCode: "validation_failed"`.

Validation issues include:

- `code`
- `path`
- `field`
- `message`
- `severity`

Example issue:

```json
{
  "code": "WARMUP_TOO_SHORT",
  "path": "settings.warmupPeriod",
  "field": "settings",
  "severity": "warning",
  "message": "Warmup setting is shorter than this strategy's inferred lookback.",
  "suggestion": "Set warmupPeriod to at least 200 for more consistent behavior when prehistory is limited or after gaps."
}
```

Recommended repair loop:

1. Resolve strategy semantics into fragments.
2. Assemble selected fragments into a full `signalGraph` draft.
3. Run local preflight; do not call remote validation while local errors remain.
4. Validate remotely.
5. Repair the payload using `code`, `path`, `message`, and `suggestion`.
6. Revalidate until blocking issues are gone.

```bash
export TRASEQ_BASE_URL="https://your-traseq.example.com"
```

## Ready-to-use assets

Two reference files are available to help you get started:

- **`traseq-agent-tool-schema.json`** — a tool contract for the public API,
  ready to use with tool-enabled AI agents.
- **`traseq-agent-example.ts`** — a minimal TypeScript client with a
  validate-repair loop helper.

The first-party agent package is `@traseq/agent`. It provides a guided
research service layer for Claude, Codex, and other MCP clients, while staying
provider-agnostic. It does not require an AI provider key.

## Claude and Codex MCP setup

For Codex, Claude Code, Claude Desktop, or any MCP-capable client, start with
the setup helper instead of hand-writing JSON. Codex example:

```sh
export TRASEQ_API_KEY="trsq_..."
npx -y --package @traseq/agent traseq-agent setup-mcp --client codex --write --probe
```

Dry-run prints the install command and MCP JSON without mutating client config:

```sh
npx -y --package @traseq/agent traseq-agent setup-mcp --client codex --probe
```

If the MCP server already exists and you want to reinstall it, add `--force`:

```sh
npx -y --package @traseq/agent traseq-agent setup-mcp --client codex --write --force --probe
```

Other clients follow the same pattern with client-appropriate flags:

```sh
npx -y --package @traseq/agent traseq-agent setup-mcp --client claude-code --write --probe
npx -y --package @traseq/agent traseq-agent setup-mcp --client claude-desktop --print-config
npx -y --package @traseq/agent traseq-agent setup-mcp --client generic --print-config
```

`--write` installs when the selected client supports it. `--probe` checks
manifest, workspace, usage, and capabilities so the agent can tell whether the
key has the scopes needed for guided research.

Use `--scope project` for shared repository config. Project scope keeps
`TRASEQ_API_KEY` as `${TRASEQ_API_KEY}` instead of writing the secret into
config. Do not paste API keys into prompts.

On shared hosts, prefer the dry-run output and edit the config manually:
during `--write`, `claude mcp add` and `codex mcp add` accept env values via
argv, so the API key is briefly visible to other users via `ps`. Claude
Desktop does not expand `${VAR}` placeholders, so its install always inlines
the key into the user-level config file — keep that file restricted to your
user. `--scope project` is not applicable to Claude Desktop and is treated as
`--scope user`.

For a one-command local fallback, `setup-mcp` also accepts `--api-key`, but
prefer `TRASEQ_API_KEY` because flag values may remain in shell history.

After setup, give the client a task-oriented prompt:

```text
Use the Traseq MCP server. First call the MCP tool `start_research_engagement` with prompt "Validate a BTCUSDT 4h strategy idea." Do not search the repo; if the tool is unavailable, tell me the Traseq MCP server is not connected.
```

The MCP server advertises `traseq_guided_research` through `prompts/list` and
orders guided tools first. The intended sequence is
`start_research_engagement` → external draft authoring →
`run_guided_research_round` → optional `summarize_research_engagement`.
Lower-level Public Agent API tools remain available for advanced automation.

## TypeScript SDK example

Here's the core pattern for building an agent client. See
`traseq-agent-example.ts` for a full working version.

```ts
const client = new TraseqPublicAgentClient(
  process.env.TRASEQ_BASE_URL!,
  process.env.TRASEQ_API_KEY!,
);

const workspace = await client.getWorkspaceContext();
const usage = await client.getUsage();
const capabilities = await client.getCapabilities();

const authoringPayload = await validateWithRepairLoop(
  client,
  {
    signalGraph,
    settings: {
      positionStyle: 'single',
      warmupPeriod: 200,
    },
  },
  async (currentPayload, context) => {
    return repairPayloadWithYourModel({
      capabilities,
      currentPayload,
      issues: context.issues,
    });
  },
);

const draft = await client.createStrategy({
  name: 'Agent-authored EMA breakout',
  description: 'Generated by an external AI agent',
  ...authoringPayload,
});

const ready = await client.finalizeStrategyVersion(draft.id, {
  version: draft.versions[0].version,
  ...authoringPayload,
});

const backtest = await client.runBacktest({
  strategyVersionId: ready.id,
  config: {
    timeframe: '1h',
    signalInstrument: { symbol: 'BTCUSDT' },
    initialBalance: 10000,
    execution: {
      feeModel: {
        kind: 'tiered_maker_taker',
        tiers: [
          {
            minCumulativeNotional: 0,
            makerRate: 0.001,
            takerRate: 0.001,
          },
        ],
      },
      slippage: {
        kind: 'fixed',
        unit: 'bps',
        value: 5,
      },
    },
  },
});
```

The backtest response includes `appLinks` for handoff into the Traseq app and
`runContext` for agent-readable evidence context. Prefer showing
`appLinks.backtest` or `appLinks.backtestCharts` to the user instead of only
printing UUIDs, and use `runContext.instrument`, `runContext.timeframe`, and
`runContext.range` when summarizing what was tested.

The key point is the strict ordering — always follow this sequence:

1. read workspace, usage, and capabilities
2. generate `signalGraph`
3. validate
4. repair
5. create
6. finalize
7. backtest

## Agent tool schema

Instead of giving your AI model raw HTTP access, wrap the API in a structured
tool layer. See `traseq-agent-tool-schema.json` for the full JSON definition.

Recommended tools:

- `start_research_engagement`
- `run_guided_research_round`
- `summarize_research_engagement`
- `get_manifest`
- `get_workspace_context`
- `get_usage`
- `get_capabilities`
- `resolve_strategy_semantics`
- `assemble_signal_graph`
- `preflight_strategy_draft`
- `validate_strategy`
- `create_strategy`
- `finalize_strategy_version`
- `run_backtest`
- `get_backtest`

`validate_strategy` uses the SDK authoring schema, not a shallow
`signalGraph: object` wrapper. The generated tool schema requires nested fields
such as `signalGraph.strategy.entry.action.sizing`,
`signalGraph.strategy.risk`, and `settings.positionStyle`; use
`assemble_signal_graph` and `preflight_strategy_draft` before calling the remote
validator.

## Suggested system prompt

Give your AI model a system prompt that enforces the capability contract and
validate-repair loop.

```text
You are an external AI strategy-authoring agent for Traseq.

Before composing any strategy:
1. Call get_workspace_context.
2. Call get_usage.
3. Call get_capabilities.
4. Use usage and capabilities as the only source of truth for current budget, limits, supported node kinds, strategy bindings, risk metadata, indicators, operators, and enums.

Authoring rules:
- Author only SignalGraph v2 payloads.
- Keep the first payload minimal and valid.
- Reuse simple, stable node ids such as close_price, ema_fast, entry_trigger.
- Use kind: "market" with field for candle data; do not use a price node.
- Use indicator, args, and optional top-level output on indicator nodes.
- Use args.length for indicator lookbacks; put source inside args.source.
- Use offset for historical bars; do not use shift.
- Bind entry.trigger to one bool ref and exits[0].when to one bool ref when an exit is present.
- Do not invent unsupported fields, args, bindings, risk paths, or operators.

Validation rules:
- Use resolve_strategy_semantics, assemble_signal_graph, and preflight_strategy_draft before validate_strategy.
- Always call validate_strategy before create_strategy or finalize_strategy_version.
- Treat issue code, path, message, and suggestion as the repair source of truth.
- Modify the minimum subtree needed to resolve each blocking issue.
- Revalidate after every repair pass.

Execution rules:
- Create drafts before finalizing.
- Call get_usage again before write-heavy or backtest-heavy workflows if the session is long-lived.
- Run backtests only after a ready version exists.
- Summarize backtest outcomes from summaryJson before inspecting large artifacts.

Error handling rules:
- When an API error includes publicAgent metadata, show its title, explanation, retryable flag, nextSteps, and app links.
- For usage or plan limits, direct the user to the billing_plan or billing link returned by Traseq.
- Do not send users to a public marketing pricing page unless Traseq did not provide an app link.
```

## Repair loop template

When validation fails, repair the specific issues rather than regenerating the
entire strategy. Here's a template for a deterministic repair loop.

```text
Inputs:
- user_intent
- current_payload
- capabilities
- blocking_issues[]

Task:
- Return a revised payload only.
- Preserve unchanged fields.
- Modify the minimum subtree needed to resolve the listed blocking issues.

Repair policy:
1. Read each issue.code and issue.path.
2. If suggestion is present, prefer it over freeform reinterpretation.
3. Do not rename unrelated node ids.
4. Do not introduce new node kinds unless capabilities allow them.
5. Keep strategy settings unless an issue directly points to settings.
```

## Tips

- **Start simple.** One trigger plus one confirmation filter is much easier to
  repair than a complex multi-branch graph.
- **Cache capabilities.** Cache the response from `GET /public/v1/capabilities`
  per workspace tier, and refresh when the subscription changes.
- **Use issue codes for repairs.** Deterministic repair rules based on
  `issue.code` are more reliable than asking the model to reinterpret error
  messages.
- **Fix errors first, warnings second.** Feed the repair loop only blocking
  issues initially — handle warnings in a second pass once the payload is
  valid.
