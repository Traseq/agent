這份指南是給第三方開發者的。目標是讓你可以用自己的 AI agent，透過
Traseq 的 public API key，完成 `validate -> create -> finalize -> backtest`
整條策略流程。

## 開始前

Public Agent API 使用的是 workspace API key，不是 wallet private key、交易所
secret，或其他資產控制私鑰。請先到 Traseq app 的 **Settings > API Keys**
建立 key，在明文只顯示一次時複製，並存進 secret manager 或 runtime
environment，例如 `TRASEQ_API_KEY`。

所有方案都可以建立 API key 並執行核心 agent workflow。Free 與 Plus 主要由較低的
key quota、research credits、strategy limits、saved-result limits 與 workspace
capacity 控制使用範圍。

不要把 API key 貼進 AI prompt，也不要讓 model 看見 token。agent host 應該在
model tool input 之外，把 key 注入 `x-api-key` HTTP header。

若 agent 需要建立策略與執行回測，建議建立 `member` key，並只授予必要 scopes：

- `workspace_read`
- `strategies_read`
- `strategies_write`
- `backtests_read`
- `backtests_write`

只有在 agent 需要讀系統模板時，才額外加入 `system_strategies_read`。

## Strategy Authoring Contract

Public Agent API 的策略寫入面只接受 **SignalGraph v2**。Agent 應送出
`signalGraph` 與 `settings`；後端可能再編譯成內部執行格式，但那些內部格式不屬於
公開 write contract。

## 最小 scopes

若你的 agent 需要自行驗證、寫入策略、並執行回測，建議最小 scopes 是：

- `workspace_read`
- `strategies_read`
- `strategies_write`
- `backtests_read`
- `backtests_write`

常見補充 scopes：

- `system_strategies_read`
  當 agent 需要從內建模板起手時很有用。

## 建議呼叫順序

1. `GET /public/v1`
   先讀 manifest，確認目前 API workflow contract。
2. `GET /public/v1/workspace`
   讀 workspace identity、API key scopes、subscription tier。
3. `GET /public/v1/usage`
   在 write 或 backtest workflow 前檢查目前 budget、usage 與 hard limits。
4. `GET /public/v1/capabilities`
   在 authoring 前先拿可用 indicator、indicator args、node fields、strategy bindings、risk authoring metadata、operator、tier limits
   與 validation issue shape。
5. `POST /public/v1/strategies/validate`
   在本地 assembly/preflight 後驗證草稿 payload，直到 `valid === true`。
6. `POST /public/v1/strategies`
   建立 draft strategy。
7. `POST /public/v1/strategies/:id/versions/finalize`
   把版本升成 `ready`。
8. `POST /public/v1/backtests`
   建立回測。
9. `GET /public/v1/backtests/:id`
   輪詢直到回測進入終態。

## Machine-readable capabilities

外部 agent 應在寫策略前先呼叫 `GET /public/v1/capabilities`。

這個回應會列出：

- 可用的 `indicators` 與 `args`
- 可用的 `signalGraph.nodes` 與 node fields
- 可用的 strategy bindings 與 risk authoring metadata
- 可用的 `operators`
- 可用 enum sets，例如 `timeframes`、`marketFields`、`patterns`
- tier-aware 的 `limits`
- validation issue shape，包括 `code`、`path`、`field`、`message`、`severity`

外部 agent 應把這個 endpoint 視為 authoring source of truth，而不是靠 prompt
自己猜 node shape、indicator args、binding、risk path 或 operator。

## 結構化 validation issues

`POST /public/v1/strategies/validate` 現在會回傳 machine-readable issues。Graph
驗證問題與 400 request schema failures 都使用相同的 flat `issues[]` shape；request
schema failures 另外會帶 `errorCode: "validation_failed"`。

Validation issue 至少包含：

- `code`
- `path`
- `field`
- `message`
- `severity`

範例：

```json
{
  "code": "WARMUP_TOO_SHORT",
  "path": "settings.warmupPeriod",
  "field": "settings",
  "severity": "warning",
  "message": "Warmup setting is shorter than this strategy's inferred lookback.",
  "suggestion": "Set warmupPeriod to at least 200 for more consistent behavior when prehistory is limited or after gaps."
}
```

建議修復 loop：

1. 把使用者意圖 resolve 成 strategy semantic fragments
2. 把選定 fragments assemble 成完整 `signalGraph` draft
3. 先跑本地 preflight；本地錯誤未清掉前不要呼叫遠端 validation
4. 呼叫遠端 validate
5. 使用 `code`、`path`、`message`、`suggestion` 修 payload
6. 重新 validate，直到 blocking issues 消失

## 可直接複製的資產

repo 內已附上兩個可直接複製的資產：

- `docs/public-docs/reference/traseq-agent-tool-schema.json`
  適合 tool-enabled agent 的薄工具層 schema。
- `docs/public-docs/reference/traseq-agent-example.ts`
  最小可用的 TypeScript client，加上 validate-repair loop helper。

這個 repo 內 agent package 使用 `@traseq/agent`。它替 Claude、Codex 與其他 MCP
client 提供 guided research service layer，同時維持 provider-agnostic；不需要
AI provider key。

## Claude 與 Codex MCP setup

不論是 Codex、Claude Code、Claude Desktop 或其他支援 MCP 的 client，建議先用
setup helper，不要手動貼 JSON 起手。以 Codex 為例：

```sh
export TRASEQ_API_KEY="trsq_..."
npx -y --package @traseq/agent traseq-agent setup-mcp --client codex --write --probe
```

Dry-run 會印出安裝指令與 MCP JSON，不會改 client config：

```sh
npx -y --package @traseq/agent traseq-agent setup-mcp --client codex --probe
```

如果 MCP server 已存在、你要重新安裝，加上 `--force`：

```sh
npx -y --package @traseq/agent traseq-agent setup-mcp --client codex --write --force --probe
```

其他 client 沿用同一套流程，依 client 特性帶不同 flags：

```sh
npx -y --package @traseq/agent traseq-agent setup-mcp --client claude-code --write --probe
npx -y --package @traseq/agent traseq-agent setup-mcp --client claude-desktop --print-config
npx -y --package @traseq/agent traseq-agent setup-mcp --client generic --print-config
```

加上 `--write` 才會在支援的 client 上寫入設定。`--probe` 會依序檢查
manifest、workspace、usage 與 capabilities，讓 agent 能判斷 API key 是否具備
guided research 需要的 scopes。

若要放進共享 repo config，使用 `--scope project`。Project scope 會保留
`${TRASEQ_API_KEY}`，不把 secret 寫入 config。不要把 API key 貼進 prompt。

在共用主機上，建議用 dry-run 輸出後手動編輯 config：`--write` 會呼叫
`claude mcp add` 或 `codex mcp add`，這些 CLI 透過 argv 接收 env 值，API key
會短暫出現在 `ps`/`/proc` 中。Claude Desktop 不會展開 `${VAR}` 佔位符，所以
它的安裝一律會把 key inline 寫進 user 層級 config 檔，務必確認該檔案不是
world-readable。`--scope project` 對 Claude Desktop 不適用，會被當成
`--scope user` 處理。

若只想用一行完成本機備援安裝，`setup-mcp` 也支援 `--api-key`，但建議優先用
`TRASEQ_API_KEY`，因為 flag 值可能留在 shell history。

安裝後，給 client 一句任務導向 prompt：

```text
Use the Traseq MCP server. First call the MCP tool `start_research_engagement` with prompt "Validate a BTCUSDT 4h strategy idea." Do not search the repo; if the tool is unavailable, tell me the Traseq MCP server is not connected.
```

MCP server 會透過 `prompts/list` 暴露 `traseq_guided_research`，並把 guided
tools 排在前面。建議流程是 `start_research_engagement` → 外部 agent author
draft → `run_guided_research_round` → 必要時再
`summarize_research_engagement`。低階 Public Agent API tools 仍保留給進階自動化。

## TypeScript SDK 範例

若你要完整可複製版本，直接使用 `docs/public-docs/reference/traseq-agent-example.ts`。
核心流程應該長這樣：

```ts
const client = new TraseqPublicAgentClient(process.env.TRASEQ_API_KEY!);

const workspace = await client.getWorkspaceContext();
const usage = await client.getUsage();
const capabilities = await client.getCapabilities();

const authoringPayload = await validateWithRepairLoop(
  client,
  {
    signalGraph,
    settings: {
      positionStyle: 'single',
      warmupPeriod: 200,
    },
  },
  async (currentPayload, context) => {
    return repairPayloadWithYourModel({
      capabilities,
      currentPayload,
      issues: context.issues,
    });
  },
);

const draft = await client.createStrategy({
  name: 'Agent-authored EMA breakout',
  description: 'Generated by an external AI agent',
  ...authoringPayload,
});

const ready = await client.finalizeStrategyVersion(draft.id, {
  version: draft.versions[0].version,
  ...authoringPayload,
});

const backtest = await client.runBacktest({
  strategyVersionId: ready.id,
  config: {
    timeframe: '1h',
    signalInstrument: { symbol: 'BTCUSDT' },
    initialBalance: 10000,
    execution: {
      feeModel: {
        kind: 'tiered_maker_taker',
        tiers: [
          {
            minCumulativeNotional: 0,
            makerRate: 0.001,
            takerRate: 0.001,
          },
        ],
      },
      slippage: {
        kind: 'fixed',
        unit: 'bps',
        value: 5,
      },
    },
  },
});
```

Backtest 回傳會包含 `appLinks` 與 `runContext`。Agent 應優先把
`appLinks.backtest` 或 `appLinks.backtestCharts` 呈現給使用者導流到 Traseq
App，而不是只輸出 UUID；摘要回測證據時則使用 `runContext.instrument`、
`runContext.timeframe`、`runContext.range` 描述實際測試條件。

真正重要的不是 wrapper class 本身，而是 SDK 強制維持這個順序：

1. 讀 workspace、usage 與 capabilities
2. 產生 `signalGraph`
3. validate
4. repair
5. create
6. finalize
7. backtest

## Agent tool schema

建議替 agent 包一層薄的 tool interface，而不是讓 model 自由拼 HTTP。完整
JSON asset 可直接使用 `docs/public-docs/reference/traseq-agent-tool-schema.json`。

建議工具：

- `start_research_engagement`
- `run_guided_research_round`
- `summarize_research_engagement`
- `get_manifest`
- `get_workspace_context`
- `get_usage`
- `get_capabilities`
- `resolve_strategy_semantics`
- `assemble_signal_graph`
- `preflight_strategy_draft`
- `validate_strategy`
- `create_strategy`
- `finalize_strategy_version`
- `run_backtest`
- `get_backtest`

`validate_strategy` 使用 SDK authoring schema，而不是淺層的
`signalGraph: object` wrapper。generated tool schema 會要求 nested fields，例如
`signalGraph.strategy.entry.action.sizing`、`signalGraph.strategy.risk` 與
`settings.positionStyle`；請先用 `assemble_signal_graph` 與
`preflight_strategy_draft`，再呼叫遠端 validator。

## 建議 system prompt

system prompt 應強制 model 尊重 capability contract 與 validate-repair loop。

```text
You are an external AI strategy-authoring agent for Traseq.

Before composing any strategy:
1. Call get_workspace_context.
2. Call get_usage.
3. Call get_capabilities.
4. Use usage and capabilities as the only source of truth for current budget, limits, supported node kinds, strategy bindings, risk metadata, indicators, operators, and enums.

Authoring rules:
- Author only SignalGraph v2 payloads.
- Keep the first payload minimal and valid.
- Reuse simple, stable node ids such as close_price, ema_fast, entry_trigger.
- Use kind: "market" with field for candle data; do not use a price node.
- Use indicator, args, and optional top-level output on indicator nodes.
- Use args.length for indicator lookbacks; put source inside args.source.
- Use offset for historical bars; do not use shift.
- Bind entry.trigger to one bool ref and exits[0].when to one bool ref when an exit is present.
- Do not invent unsupported fields, args, bindings, risk paths, or operators.

Validation rules:
- Use resolve_strategy_semantics, assemble_signal_graph, and preflight_strategy_draft before validate_strategy.
- Always call validate_strategy before create_strategy or finalize_strategy_version.
- Treat issue code, path, message, and suggestion as the repair source of truth.
- Modify the minimum subtree needed to resolve each blocking issue.
- Revalidate after every repair pass.

Execution rules:
- Create drafts before finalizing.
- Call get_usage again before write-heavy or backtest-heavy workflows if the session is long-lived.
- Run backtests only after a ready version exists.
- Summarize backtest outcomes from summaryJson before inspecting large artifacts.

Error handling rules:
- When an API error includes publicAgent metadata, show its title, explanation, retryable flag, nextSteps, and app links.
- For usage or plan limits, direct the user to the billing_plan or billing link returned by Traseq.
- Do not send users to a public marketing pricing page unless Traseq did not provide an app link.
```

## Repair loop 模板

不要每次都要求 model 重新生成整份策略。建議使用 deterministic repair
loop。

```text
Inputs:
- user_intent
- current_payload
- capabilities
- blocking_issues[]

Task:
- Return a revised payload only.
- Preserve unchanged fields.
- Modify the minimum subtree needed to resolve the listed blocking issues.

Repair policy:
1. Read each issue.code and issue.path.
2. If suggestion is present, prefer it over freeform reinterpretation.
3. Do not rename unrelated node ids.
4. Do not introduce new node kinds unless capabilities allow them.
5. Keep strategy settings unless an issue directly points to settings.
```

## 實務建議

- 第一版 graph 先做小。單一 trigger 加單一 filter 比大而複雜的 graph
  更容易修。
- `GET /public/v1/capabilities` 可以依 workspace tier 快取，但 workspace
  或 subscription 改變時要刷新。
- 修復策略時，優先根據 `issue.code` 寫 deterministic repair rule，不要每次都
  讓 model 重新解讀同一段錯誤文案。
- 第一輪先只處理 blocking issues；等 payload 有效後，再決定要不要處理 warnings。
